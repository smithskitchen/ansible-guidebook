1. What is Ansible?

Ansible is an open-source, agentless IT automation software. Thousands of organizations use it for:

  • Configuration Management: Install software and apply system settings consistently across servers.
  • Application Deployment: Automate the installation and updating of applications.
  • Orchestration: Coordinate processes/multiple servers, like rolling upgrades or cluster setups.
  • Provisioning: Set up virtual machines and cloud infrastructure.
Why Ansible? It is simple, secure, agentless (uses SSH), powerful, and has a huge ecosystem.
+--------------------+ | Control Node | | (Ansible Installed)| +---------+----------+ | SSH/WinRM | Connections v +------------+-------------+ | Managed Nodes (Servers) | <--- (No agent required!) +-------------------------+
  • Control Node: Where you run Ansible commands/playbooks (your system or a management server).
  • Managed Nodes: The servers/devices you automate; you only need SSH (Linux/Unix) or WinRM (Windows).

2. Understanding Ansible’s Core Concepts

  • Control Node: The machine where you install and run Ansible. Can be your laptop, a jump host, or a CI/CD runner.
  • Managed Node: The remote servers or devices (on-prem or cloud) where Ansible performs automation tasks.
  • Inventory File ("hosts"): A text file (INI or YAML) listing your managed servers, organized into groups (e.g., web, db, prod). Ansible uses the inventory to know which servers to automate.
  • Modules: The plugins that actually perform the work (e.g., install software, copy files, run commands).
  • Playbooks: Human-readable files (in YAML) that define what tasks to run (using modules), in what order, and against which groups.
Biggest Advantages:
Agentless (uses SSH), simple language (YAML), powerful for both simple and complex automation needs.

3. YAML Basics for Ansible

YAML (YAML Ain't Markup Language) is the only language for writing Ansible playbooks. It is designed to be easy for non-programmers and programmers alike:

  • Indentation matters! Use spaces (never tabs), usually 2 spaces per level.
  • Lists use a dash -
  • Key-Value Pairs use a colon key: value
  • You can nest lists/dictionaries for complex structures.
# Minimal YAML playbook example:
---
- name: Ping all hosts
  hosts: all
  tasks:
    - name: Test connectivity
      ping:
Explanation:
  • name: is a label for the play.
  • hosts: all means run on every host known in inventory.
  • tasks: is a list of things to do, here just a ping test.
Useful YAML resources: YAML in Ansible, YAML Official, YAMLLint

4. Installing Ansible (Quick Start)

On Ubuntu/Debian

sudo apt update
sudo apt install software-properties-common -y
sudo add-apt-repository --yes --update ppa:ansible/ansible
sudo apt install ansible -y

On Red Hat/CentOS/Rocky/Oracle Linux

sudo dnf install epel-release -y
sudo dnf install ansible -y

With pip (works everywhere Python is installed)

pip install --user ansible
Test install with: ansible --version

5. Setting Up SSH Access (Essential for Ansible!)

  1. Generate an SSH Key (if you don't already have one):
    ssh-keygen -t rsa -b 4096
    (Accept defaults by pressing Enter)
  2. Copy the SSH public key to each managed node (the server you want to automate), for example:
    ssh-copy-id user@192.168.1.100
    (Replace user and 192.168.1.100 appropriately)
+-------------+ SSH +----------------+ | Control +-----------------> Managed Node | | Node | (public key) | (no agent!) | +-------------+ +----------------+
You can manage localhost for testing: Ansible doesn't need SSH for localhost if ansible_connection=local is set!

6. Creating Your First Inventory File

INI Format Example:

[local]
localhost ansible_connection=local

[web]
192.168.1.10
192.168.1.11

YAML Format Example:

all:
  hosts:
    localhost:
      ansible_connection: local
    web1:
      ansible_host: 192.168.1.10
    web2:
      ansible_host: 192.168.1.11
Tip: Save this as hosts (INI) or hosts.yaml in your project directory. Use -i hosts or -i hosts.yaml when running Ansible.

7. Verifying Connections with the Ping Module

Now, test connectivity from your control node to all hosts in your inventory: 

ansible all -i hosts -m ping

Expected Output: 

localhost | SUCCESS => { "changed": false, "ping": "pong" }
192.168.1.10 | SUCCESS => { "changed": false, "ping": "pong" }
192.168.1.11 | SUCCESS => { "changed": false, "ping": "pong" }
If you see SUCCESS for each host: you’re ready to automate!

If it does not work:
  • Check your inventory file (-i hosts)
  • Test SSH with ssh user@host
  • Make sure firewall/security group allows SSH!
  • Verify ansible_user and ansible_host settings for non-default usernames/hosts

8. Practical Exercises

  • Install Ansible on your workstation or VM.
  • Set up SSH key-based access from your control node to at least one managed node or localhost.
  • Create a simple hosts inventory file listing your server(s).
  • Run ansible all -i hosts -m ping and confirm successful output.
  • Optional: Try editing the inventory to add more hosts, test SSH, and explore ansible -m command -a "uptime".
